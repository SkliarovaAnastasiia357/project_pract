# Описание тестов проекта

## Что это за тесты

В проекте есть 2 группы автоматических тестов:

- frontend-тесты на `TypeScript`
- backend-тесты на `Go`

Они нужны, чтобы быстро проверить основную логику проекта без ручного кликанья по интерфейсу.

Проще говоря, тесты проверяют:

- регистрацию и вход
- ошибки в формах
- доступ к защищённым разделам
- маршруты приложения
- работу mock API
- базовую логику профиля и проектов

Важно: это не e2e-тесты. Они не открывают браузер и не проверяют живой сервер. Здесь тестируются функции и модули напрямую.

---

## Как запускать

```powershell
npm run test:frontend
npm run test:backend
npm test
```

Что делает каждая команда:

- `npm run test:frontend` запускает frontend-тесты
- `npm run test:backend` запускает backend-тесты через `go test ./...`
- `npm test` запускает всё сразу

Перед коммитом также срабатывает `.githooks/pre-commit`, который автоматически запускает:

```powershell
npm test
```

Если какой-то тест падает, коммит отменяется.

Для запуска нужно:

- `Node.js` для frontend
- `Go` для backend

---

## Как устроены frontend-тесты

Frontend-тесты запускаются через файлы:

- `tests/frontend/run-frontend-tests.mjs`
- `tests/frontend/run-frontend-tests.ts`

Здесь используется простой собственный раннер:

- тесты вызываются как обычные функции
- проверки делаются через `node:assert/strict`
- если условие не совпало, тест завершается ошибкой

Это проще, чем Jest или Vitest, но для текущего MVP этого достаточно.

## Что проверяет frontend

### `authMvp.test.ts`

Проверяет логику авторизации и базового поведения приложения.

Основные проверки:

- успешная регистрация
- ошибка при повторном email
- ошибка при некорректном email
- ошибка при коротком пароле
- ошибка при несовпадении паролей
- успешный вход
- ошибка при неверном пароле
- logout
- запрет доступа после выхода
- меню навигации
- данные профиля

Какие функции проверяются:

- `validateRegistrationForm`
- `registerUser`
- `loginUser`
- `logoutUser`
- `canAccessProtectedPage`
- `resolveProtectedPageAccess`
- `getNavigationMenu`
- `getProfilePageData`

### `appRoutes.test.ts`

Проверяет список маршрутов приложения.

Тест защищает проект от ситуации, когда какой-то путь случайно удалили или переименовали.

Проверяемый файл:

- `src/app/routePaths.ts`

### `mockApi.test.ts`

Проверяет mock API, то есть имитацию backend на стороне frontend.

Проще говоря, этот тест показывает, что frontend может работать даже без реального сервера.

Основные проверки:

- регистрация пользователя
- login/logout
- получение и изменение профиля
- добавление и удаление навыков
- создание, чтение, изменение и удаление проекта
- ошибки валидации
- ошибка доступа после logout

Какие функции проверяются:

- `mockApi.register`
- `mockApi.login`
- `mockApi.logout`
- `mockApi.getProfile`
- `mockApi.updateProfile`
- `mockApi.addSkill`
- `mockApi.deleteSkill`
- `mockApi.listProjects`
- `mockApi.createProject`
- `mockApi.getProject`
- `mockApi.updateProject`
- `mockApi.deleteProject`

### `workspaceSummary.test.ts`

Проверяет сводку профиля.

Проще говоря, тест смотрит, правильно ли считается, насколько профиль заполнен.

Основные проверки:

- пустой профиль даёт 0%
- заполненный профиль даёт 100%
- корректно считаются карточки навыков и проектов

Проверяемая функция:

- `getWorkspaceSummary`

## Что проверяет backend

Backend-тесты лежат в:

- `tests/backend/auth_service_test.go`

Они запускаются так:

```powershell
go test ./...
```

Проверяемый пакет:

- `src/backend/auth`

Основные функции:

- `RegisterUser`
- `ValidateRegistration`
- `LoginUser`
- `LogoutUser`
- `CanAccessProtectedEndpoint`
- `GetCurrentUser`

Что именно проверяется:

- успешная регистрация
- создание пользователя с хешированным паролем
- ошибка при повторном email
- ошибки валидации
- успешный login
- ошибка при неверных данных
- потеря доступа после logout
- проверка доступа к защищённым функциям
- получение текущего пользователя по сессии

Самый важный момент backend-тестов: они проверяют, что пароль хранится не в открытом виде, а в виде хеша.

## Какие методы тестирования используются

В проекте используются:

- позитивные сценарии
- негативные сценарии
- проверка валидации
- проверка бизнес-логики без UI
- проверка mock API
- проверка авторизации и доступа

Проще говоря:

- unit-тесты проверяют отдельные функции
- часть frontend-проверок похожа на integration-тесты модулей, потому что там тестируется связанный сценарий целиком

## GitHub Actions CI/CD

В проекте настроен GitHub Actions workflow:

- `.github/workflows/tests.yml`

Что он делает:

- запускается при `push` в `main` и `master`
- запускается при создании и обновлении `pull request`
- поднимает среду `ubuntu-latest`
- устанавливает `Node.js 22`
- устанавливает `Go 1.22`
- запускает frontend-тесты
- запускает backend-тесты

Проще говоря, GitHub сам автоматически проверяет, что проект не сломался после новых изменений.

Что здесь относится к CI:

- автоматический запуск тестов после изменений в репозитории
- единая проверка в одинаковом окружении
- быстрый сигнал, что после push или PR всё проходит или что-то сломано

Что здесь пока не относится к CD:

- нет автоматического деплоя
- нет автоматической публикации сборки на сервер

То есть сейчас в проекте точно есть `CI`, а `CD` в классическом смысле пока не настроен.

Если нужно коротко сказать на предзащите:

> В проекте настроен GitHub Actions CI. После push и pull request автоматически поднимается окружение, ставятся Node.js и Go, а затем запускаются frontend и backend тесты. Это позволяет сразу видеть, не сломали ли новые изменения проект.

## Что можно сказать на предзащите

Короткий ответ:

> В проекте есть frontend и backend автотесты. Frontend-тесты проверяют авторизацию, маршруты, mock API, профиль и проекты. Backend-тесты проверяют регистрацию, логин, logout, валидацию и защищённый доступ. Тесты запускаются командами `npm run test:frontend`, `npm run test:backend` и автоматически через `pre-commit`.

Если спросят, почему не e2e:

> Сейчас мы тестируем основную бизнес-логику MVP. Это быстрее и помогает ловить ошибки в ключевых сценариях. Полноценные e2e-тесты можно добавить следующим этапом.

Если спросят, что покрыто лучше всего:

> Лучше всего покрыты регистрация, авторизация, валидация форм, mock API, работа профиля и проектов, а также backend-логика сессий и доступа.

## Ограничения

Сейчас тесты не проверяют:

- реальный браузерный сценарий от начала до конца
- живой backend по HTTP
- базу данных
- нагрузку
- внешний вид интерфейса

Это нормально для текущего этапа, потому что уже покрыта основная логика MVP.

---

## Краткий пересказ

В проекте есть frontend и backend автотесты. Frontend проверяет регистрацию, вход, маршруты, mock API, профиль и проекты. Backend проверяет регистрацию, login, logout, валидацию, сессии и доступ к защищённым функциям.

Тесты нужны для быстрой проверки важной логики проекта. Они не запускают браузер и не тестируют живой сервер, но хорошо защищают MVP от типичных ошибок и регрессий.
