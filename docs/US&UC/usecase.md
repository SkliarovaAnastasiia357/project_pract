# Use Case документация — MVP

## UC‑1: Регистрация пользователя

| Поле | Значение |
|------|----------|
| **Вариант использования** | Регистрация нового пользователя |
| **Контекст использования** | Пользователь впервые заходит на платформу и хочет создать аккаунт |
| **Область действия** | Цифровая платформа для подбора участников и формирования команд |
| **Уровень** | Цель пользователя |
| **Основное действующее лицо (актор)** | Новый пользователь (неавторизованный) |
| **Участники (дополнительные действующие лица)** | Система |
| **Предусловие** | Пользователь открыл сайт, не авторизован |
| **Минимальные гарантии успеха** | Система создаёт запись пользователя, но профиль может быть не заполнен |
| **Гарантии успеха** | Пользователь авторизован, перенаправлен на заполнение профиля, пароль захэширован |
| **Триггер** | Пользователь нажимает «Регистрация» и заполняет форму |
| **Описание (основной сценарий)** | 1. Система отображает форму регистрации (Имя, Email, Пароль). 2. Пользователь вводит данные и нажимает «Зарегистрироваться». 3. Система проверяет уникальность Email. 4. Система создаёт пользователя, хеширует пароль. 5. Система автоматически авторизует пользователя. 6. Система перенаправляет на страницу «Заполнение профиля». |
| **Расширения (альтернативные сценарии)** | *3a. Email уже существует → система показывает «Пользователь с таким email уже зарегистрирован». 2a. Пароль короче 6 символов → система показывает ошибку валидации. 4a. Ошибка БД → система показывает «Не удалось создать аккаунт, попробуйте позже».* |
| **Изменения в технологиях и данных** | В таблицу `users` добавляется запись: `id`, `name`, `email`, `password_hash`, `created_at`. Создаётся JWT-токен или сессия. |

## UC‑2: Вход в аккаунт

| Поле | Значение |
|------|----------|
| **Вариант использования** | Вход зарегистрированного пользователя |
| **Контекст использования** | Пользователь уже имеет аккаунт и хочет войти |
| **Область действия** | Цифровая платформа для подбора участников и формирования команд |
| **Уровень** | Цель пользователя |
| **Основное действующее лицо (актор)** | Зарегистрированный пользователь |
| **Участники (дополнительные действующие лица)** | Система |
| **Предусловие** | Пользователь не авторизован, аккаунт существует |
| **Минимальные гарантии успеха** | Система проверяет учётные данные |
| **Гарантии успеха** | Пользователь получает доступ к личному кабинету и защищённым разделам |
| **Триггер** | Пользователь нажимает «Войти» и вводит Email/пароль |
| **Описание (основной сценарий)** | 1. Система отображает форму входа. 2. Пользователь вводит Email и пароль, нажимает «Войти». 3. Система находит пользователя по Email. 4. Система проверяет пароль (хеш совпадает). 5. Система создаёт сессию / выдаёт токен. 6. Система перенаправляет на ленту проектов. |
| **Расширения (альтернативные сценарии)** | *3a. Email не найден → система показывает «Неверный email или пароль». 4a. Пароль не совпадает → система показывает «Неверный email или пароль».* |
| **Изменения в технологиях и данных** | Чтение из таблицы `users`, генерация JWT (expires_in). Запись `last_login_at` (опционально). |

## UC‑3: Заполнение и редактирование профиля

| Поле | Значение |
|------|----------|
| **Вариант использования** | Управление профилем пользователя |
| **Контекст использования** | Пользователь хочет указать свои навыки и описание, чтобы его находили капитаны проектов |
| **Область действия** | Цифровая платформа для подбора участников и формирования команд |
| **Уровень** | Цель пользователя |
| **Основное действующее лицо (актор)** | Авторизованный пользователь |
| **Участники (дополнительные действующие лица)** | Система, другие пользователи (только чтение) |
| **Предусловие** | Пользователь авторизован |
| **Минимальные гарантии успеха** | Данные профиля сохраняются в БД |
| **Гарантии успеха** | Другие пользователи видят актуальные навыки и описание при поиске |
| **Триггер** | Пользователь переходит в раздел «Мой профиль» и нажимает «Редактировать» |
| **Описание (основной сценарий)** | 1. Система отображает текущий профиль (имя, описание, список навыков). 2. Пользователь нажимает «Редактировать». 3. Система открывает форму с полями: «Описание» (textarea), блок «Навыки» (чекбоксы/теги). 4. Пользователь изменяет описание, добавляет/удаляет навыки. 5. Пользователь нажимает «Сохранить». 6. Система валидирует и обновляет данные. 7. Система показывает обновлённый профиль. |
| **Расширения (альтернативные сценарии)** | *4a. Пользователь добавляет кастомный навык (не из списка) → система сохраняет его как новый тег (если разрешено в MVP). 6a. Описание превышает 1000 символов → система показывает ошибку «Максимум 1000 символов». 6b. Ошибка БД → «Не удалось сохранить профиль».* |
| **Изменения в технологиях и данных** | Таблица `profiles` (`user_id`, `description`). Таблица `user_skills` (`user_id`, `skill_name`). API PATCH `/api/profile`. |

## UC‑4: Создание и редактирование проекта

| Поле | Значение |
|------|----------|
| **Вариант использования** | Управление карточкой проекта (CRUD) |
| **Контекст использования** | Капитан хочет опубликовать проект и набрать команду |
| **Область действия** | Цифровая платформа для подбора участников и формирования команд |
| **Уровень** | Цель пользователя |
| **Основное действующее лицо (актор)** | Авторизованный пользователь (капитан проекта) |
| **Участники (дополнительные действующие лица)** | Система, другие пользователи (только чтение) |
| **Предусловие** | Пользователь авторизован |
| **Минимальные гарантии успеха** | Проект сохраняется в БД со статусом «активен» |
| **Гарантии успеха** | Проект виден в поиске, другие пользователи могут подавать заявки |
| **Триггер** | Пользователь нажимает «Создать проект» или «Редактировать» на своей карточке |
| **Описание (основной сценарий)** | 1. Система отображает форму проекта (Название, Идея, Технологический стек, Роли для найма). 2. Пользователь заполняет поля. 3. Пользователь нажимает «Сохранить». 4. Система создаёт/обновляет запись проекта. 5. Система перенаправляет на публичную страницу проекта `/project/{id}`. 6. Система добавляет проект в раздел «Мои проекты» у автора. |
| **Расширения (альтернативные сценарии)** | *2a. Название или идея пустые → система показывает ошибку валидации. 4a. Технологический стек содержит недопустимые символы → система очищает или отклоняет. 5a. При редактировании — изменения сразу видны на странице проекта.* |
| **Изменения в технологиях и данных** | Таблица `projects` (`id`, `owner_id`, `name`, `description`, `created_at`, `updated_at`). Таблица `project_skills`, `project_roles`. API POST/PUT `/api/projects`. |

## UC‑5: Поиск проектов и пользователей по навыкам и ключевым словам

| Поле | Значение |
|------|----------|
| **Вариант использования** | Поиск с фильтрацией по навыкам (строгое совпадение) и ключевым словам |
| **Контекст использования** | Пользователь хочет найти проекты или людей с нужным стеком |
| **Область действия** | Цифровая платформа для подбора участников и формирования команд |
| **Уровень** | Цель пользователя |
| **Основное действующее лицо (актор)** | Любой пользователь (авторизованный или гость) |
| **Участники (дополнительные действующие лица)** | Система |
| **Предусловие** | В базе есть хотя бы один проект / профиль |
| **Минимальные гарантии успеха** | Система возвращает массив результатов (возможно пустой) |
| **Гарантии успеха** | Результаты отфильтрованы строго по выбранным навыкам + поиск по тексту |
| **Триггер** | Пользователь вводит текст в поисковую строку, выбирает фильтры и нажимает «Найти» |
| **Описание (основной сценарий)** | 1. Система отображает поисковую строку и фильтры (вкладка «Проекты» / «Люди», выбор навыков). 2. Пользователь вводит ключевое слово «анализ» и выбирает навык «Python». 3. Пользователь нажимает «Найти». 4. Система выполняет поиск: 4.1. Для проектов: название или описание содержит «анализ» **И** в технологическом стеке есть «Python». 4.2. Для людей: описание профиля содержит «анализ» **И** в навыках есть «Python». 5. Система возвращает карточки с результатами (название, краткое описание, список навыков/ролей). 6. Пользователь кликает на карточку — переходит на детальную страницу. |
| **Расширения (альтернативные сценарии)** | *4a. Ничего не найдено → система показывает «Ничего не найдено, попробуйте другие ключевые слова или навыки». 2a. Пользователь не выбрал навыки → поиск только по ключевым словам. 4b. Выбрано несколько навыков («Python» и «SQL») → строгое пересечение (должны быть оба).* |
| **Изменения в технологиях и данных** | API GET `/api/search?q=анализ&skills=Python&type=projects`. Индексы в БД по `name`, `description`, `skills`. |

## UC‑6: Подача и обработка заявки на участие в проекте

| Поле | Значение |
|------|----------|
| **Вариант использования** | Заявка на роль в проекте и её рассмотрение капитаном |
| **Контекст использования** | Пользователь нашёл подходящий проект и хочет присоединиться, а капитан управляет входящими заявками |
| **Область действия** | Цифровая платформа для подбора участников и формирования команд |
| **Уровень** | Цель пользователя |
| **Основное действующее лицо (актор)** | Соискатель (подача заявки), Капитан (обработка) |
| **Участники (дополнительные действующие лица)** | Система |
| **Предусловие** | 1. Соискатель авторизован и не подавал заявку на эту роль в этом проекте. 2. Проект существует и роль открыта. 3. Капитан авторизован для обработки. |
| **Минимальные гарантии успеха** | Заявка сохраняется со статусом «pending» |
| **Гарантии успеха** | Соискатель видит статус заявки. Капитан может принять/отклонить. |
| **Триггер (подача)** | Соискатель нажимает «Подать заявку» на странице проекта |
| **Описание (основной сценарий подачи)** | 1. Соискатель открывает страницу проекта. 2. Видит список ролей и кнопку «Подать заявку» напротив нужной. 3. Нажимает кнопку. 4. Система проверяет, нет ли уже заявки от этого пользователя на эту роль. 5. Система создаёт заявку со статусом «На рассмотрении». 6. Система показывает «Заявка отправлена». 7. Система добавляет заявку в раздел капитана «Входящие заявки». |
| **Описание (основной сценарий обработки)** | 1. Капитан открывает раздел «Заявки на мой проект».2. Видит список заявок с ФИО, навыками, ролью и датой. 3. Нажимает «Принять» или «Отклонить». 4. Система обновляет статус заявки. 5. Соискатель видит обновлённый статус в своём разделе «Мои заявки». |
| **Расширения (альтернативные сценарии)** | @4a (подача). Заявка уже существует → система показывает «Вы уже подавали заявку на эту роль». 3a (обработка). Ошибка БД при обновлении статуса → система показывает ошибку и не меняет статус. 6a (подача). Роль уже занята → кнопка неактивна, сообщение «Роль закрыта».* |
| **Изменения в технологиях и данных** | Таблица `applications` (`id`, `project_id`, `user_id`, `role_name`, `status`, `created_at`, `updated_at`). API POST `/api/projects/{id}/apply`, API PATCH `/api/applications/{id}/status`. |
