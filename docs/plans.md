# Teamnova Sprint 5 Finalization Plan

Дата: 2026-05-16
Период спринта: 15.05.2026–28.05.2026

## Scope

Довести текущий full-stack MVP Teamnova до финальной защиты: комплексное тестирование, исправление найденных багов, точечный рефакторинг, UI/UX-полировка, готовность к публикации, актуальная документация и финальная презентация.

## Assumptions

- Текущая рабочая ветка `sprint4-search-applications` содержит актуальный MVP Спринта 4.
- Финальный PR должен идти в production-ready базу репозитория. Если отдельной `prod`/`production` ветки нет, базой считается `main`.
- Публичная ссылка в документации: `https://teamnova.tw1.su`. Проверка ДНС/хостинга зависит от внешней инфраструктуры и может требовать ручного доступа.
- Новые runtime-зависимости не добавляются без отдельного согласования.
- Backend-архитектура, схема БД и auth/security model не меняются в рамках этого финального прохода.

## Milestone 1: QA Baseline And Bug Fixes `[x]`

Goal: получить фактическую картину качества и закрыть баги, мешающие демо.

Tasks:
- Прогнать текущие frontend/backend/build/lint проверки.
- Провести browser smoke в mock-режиме для регистрации, профиля, проекта, поиска и заявок.
- Исправить проверенный баг восстановления mock-сессии после прямого URL/перезагрузки.
- Добавить regression-тест, который падает до исправления и проходит после.

Definition of done:
- Mock-демо не теряет авторизованного пользователя при восстановлении приложения из локального состояния.
- Проверки проходят или имеют документированный инфраструктурный skip.
- Найденные баги описаны в `docs/status.md`.

Validation:
- `npm run test:frontend`
- `npm run test:backend`
- `npm run build`
- `npm run lint`
- Browser smoke на локальном Vite mock URL.

## Milestone 2: Sprint 5 UI/UX Polish `[x]`

Goal: убрать устаревшие формулировки и привести интерфейс к финальному спринту.

Tasks:
- Обновить главную страницу: заголовки, описание, sidebar и доску задач под Спринт 5.
- Убрать черновые англоязычные eyebrow-тексты из пользовательских экранов.
- Сохранить существующие рабочие сценарии auth/profile/projects/search/applications.
- Проверить адаптивность основных экранов через browser smoke.

Definition of done:
- UI не ссылается на Спринт 4 как на текущий финальный статус.
- Тексты выглядят как финальный продукт и понятны на защите.
- Нет регрессий в frontend-тестах.

Validation:
- `npm run test:frontend`
- `npm run build`
- Browser smoke `/home`, `/profile`, `/projects/new`, `/search`, `/requests`.

## Milestone 3: Documentation And Release Readiness `[x]`

Goal: синхронизировать документацию с финальным состоянием сайта и публикации.

Tasks:
- Обновить README под Спринт 5, финальную готовность, QA и деплой.
- Обновить тест-план, статус, чек-листы и модульный прогресс.
- Подготовить release/deploy checklist для хостинга.
- Зафиксировать ограничение внешнего ДНС/хостинга, если оно не проверяется из локальной среды.

Definition of done:
- Документация описывает актуальный код, команды и финальный демо-сценарий.
- Есть понятный путь публикации финальной версии на хостинге.

Validation:
- `npm run build`
- Сверка ссылок/команд в README и test-plan.

## Milestone 4: Final Presentation `[x]`

Goal: подготовить редактируемую финальную презентацию для защиты.

Tasks:
- Собрать narrative: проблема, решение, аудитория, MVP, архитектура, пользовательский сценарий, QA, деплой, итоги.
- Создать `.pptx` в `docs/presentation/`.
- Визуально проверить экспорт/рендер слайдов.
- Синхронизировать факты презентации с README и кодом.

Definition of done:
- Финальный `.pptx` лежит в репозитории.
- Слайды не содержат выдуманных метрик и соответствуют фактическому MVP.

Validation:
- Render/preview deck pages.
- Проверка содержания против README и текущего UI.

## Milestone 5: Publish To GitHub `[~]`

Goal: после зеленого gate опубликовать изменения.

Tasks:
- Проверить `git diff` и исключить случайные/чужие изменения.
- Сделать атомарный commit.
- Push текущей ветки в `origin`.
- Создать draft PR в production-ready базу (`prod`/`production`, если есть; иначе `main`).

Definition of done:
- Есть commit, remote branch и draft PR.
- PR body содержит изменения, impact и validation.

Validation:
- `git status --short`
- `gh auth status`
- `git push -u origin <branch>`
- `gh pr create --draft ...`

## Stop And Fix Rule

Если любой runnable validation command падает, следующий шаг только диагностика и исправление причины. Новая полировка, документация или презентация не продолжаются, пока текущий gate не понятен.
